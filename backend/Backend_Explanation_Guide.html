What is a Backend?

Understanding the big picture before diving into code.

💡 Simple Definition
Think of the backend as a restaurant kitchen. The mobile app (frontend) is the customer and the waiter. The customer tells the waiter what they want. The waiter sends the order to the kitchen (backend). The kitchen cooks the food (processes data) and sends it back. The customer never sees the kitchen — they only get the final result.

🏗️ What Our Backend Does

🔐
Authentication

Handles user sign up, login, and making sure only logged-in users can access data. Uses JWT tokens to verify identity.

💾
Data Management

Saves, reads, updates, and deletes financial data like transactions, budgets, and savings goals in MongoDB.

🤖
AI Processing

Sends data to OpenAI to auto-categorize expenses, generate financial insights, and predict future budgets.

⏰
Automation

Runs background jobs on a schedule — like sending budget alerts at 9 AM daily without any user action.

📌 Section 3

How a Request Flows

Let's trace exactly what happens when the mobile app sends POST /api/v1/transactions to create a new ₹500 coffee expense.

1

📱 Mobile App sends an HTTP Request

The mobile app sends a POST request to http://localhost:5000/api/v1/transactions with the transaction data in the request body: { account, type: "expense", amount: 500, description: "Coffee" }

↓
2

🚀 server.js receives the connection

The HTTP server (created in server.js) receives the incoming request and passes it to the Express application defined in app.js.

↓
3

🌐 app.js runs Global Middleware

Before reaching any route, the request passes through: CORS (is this app allowed to connect?), express.json() (parse the JSON body), rateLimiter (is this user sending too many requests?), and logger (record this request in the console).

↓
4

🗺️ transaction.routes.js — The Traffic Director

The URL /api/v1/transactions with method POST matches the route: router.post("/", authMiddleware, asyncHandler(createTransaction)). Express now knows exactly what to do.

↓
5

🔐 auth.middleware.js — The Security Guard

Before the controller can run, the authMiddleware checks the request's Authorization: Bearer eyJhbG... header. It verifies the JWT token. If invalid or missing, it immediately returns a 401 Unauthorized error. If valid, it attaches req.user = { _id, email } and calls next().

↓
6

🧠 transaction.controller.js — The Brain

Step A: Joi validates all incoming data. Is amount a positive number? Is account a valid ID?
Step B: Finds the Account in MongoDB. Calls AI service to categorize "Coffee".
Step C: Creates the Transaction in MongoDB (the model).
Step D: Deducts ₹500 from the Account balance.
Step E: Checks if this expense exceeded any Budget and sends an alert if needed.

↓
7

🗄️ Transaction Model talks to MongoDB

Transaction.create({...}) inserts a new document into the MongoDB transactions collection and returns the saved document with its new _id.

↓
8

✅ Response sent back to Mobile App

The controller calls sendSuccess(res, 201, "Transaction created", transaction) which sends a clean JSON response: { success: true, data: { transaction }, message: "Transaction created" }

📌 Section 4

The 5 Layers Explained

Each layer has one clear responsibility. This keeps the code clean and easy to maintain.

🗄️ Layer 1 — Models (src/models/)

What it is:
The database blueprint. Every model defines the exact shape, rules, and data types for one collection in MongoDB.

Think of a Model like a filled-out job application form. The form has defined fields (Name, Email, Phone). You cannot put your phone number in the Name field. The Model enforces this structure.

// Transaction.js — tells MongoDB what a Transaction must look like
const transactionSchema = new Schema({
  user:    { type: Schema.Types.ObjectId, required: true }, // who owns it
  amount:  { type: Number, min: 0.01, required: true },    // must be positive
  type:    { type: String, enum: ["income", "expense"] },    // only these 2 values
});

🛡️ Layer 2 — Middleware (src/middleware/)

What it is:
Functions that run before the controller. They check, process, or block the request.

Middleware is like airport security. Your bag (the request) must go through screening (middleware) before you can board (reach the controller).

rateLimiter.middleware.jsLimit requests per IPReturns 429 Too Many Requests
logger.middleware.jsLog every request to consoleDoes not fail — always calls next()

🗺️ Layer 3 — Routes (src/routes/)

What it is:
The URL map. Routes define which function (controller) handles which URL and HTTP method. 
// transaction.routes.js — Maps URLs to Controller functions
router.get("/",               authMiddleware, asyncHandler(getTransactions));
router.post("/",              authMiddleware, asyncHandler(createTransaction));
router.get("/:transactionId", authMiddleware, asyncHandler(getTransaction));
router.patch("/:transactionId",authMiddleware, asyncHandler(updateTransaction));
router.delete("/:transactionId",authMiddleware, asyncHandler(deleteTransaction));

🧠 Layer 4 — Controllers (src/controllers/)

What it is:
The actual business logic. Controllers validate data, talk to models and services, and send the final response.

Each controller function follows this exact pattern:

export const createTransaction = async (req, res) => {
  // Step 1: Validate input using Joi
  const { error, value } = schema.validate(req.body);
  if (error) return sendError(res, 400, "Validation failed");

  // Step 2: Do database operations
  const transaction = await Transaction.create(value);

  // Step 3: Send response
  return sendSuccess(res, 201, "Transaction created", transaction);
};

⚙️ Layer 5 — Services (src/services/)

What it is:
Specialists for complex tasks. Controllers call services when they need heavy lifting done.

If a controller needs to send an email, run AI analysis, or scan a receipt — it delegates that task to a service. This keeps controllers clean and focused on their one job: handling the HTTP request.

📌 Section 5

All Keywords Explained

Every technical term used in this backend — explained in plain English.

JWT
JSON Web Token
Authentication

A small, secure piece of text (token) given to the user after login. The mobile app sends this token with every request inside the header. The backend reads it to know who is making the request — without hitting the database every time. It has 3 parts separated by dots: Header.Payload.Signature. The Signature makes it impossible to fake.

// What a JWT looks like (eyJ = base64 encoded JSON)
eyJhbGciOiJIUzI1NiJ9.eyJfaWQiOiI2NjcifQ.xZ8k_mNqR...
bcryptjs
Password Hashing Library
Security

When a user signs up, we never save their plain password like "password123". bcryptjs converts it into an unreadable string called a hash: $2b$10$X7Vx8xK.... Even if hackers steal the database, they cannot read any passwords. When the user logs in, bcryptjs compares the typed password against the stored hash.

Joi
Data Validation Library
Validation

A library that defines and enforces rules for incoming data. We define a "schema" (rulebook) and then validate the request body against it. If anything fails the rules, Joi returns descriptive error messages automatically. It replaces hundreds of manual if/else checks.

const schema = Joi.object({
  email: Joi.string().email().required(),     // must be valid email
  amount: Joi.number().positive().required(),  // must be positive number
});
Mongoose Schema
MongoDB Document Structure Definition
Database

A Schema is the blueprint (plan) for a MongoDB document. It defines what fields exist, their data types, whether they are required, and their default values. Mongoose uses this blueprint to enforce consistency in your database.

async / await
Asynchronous Programming Syntax
JavaScript

Database queries and API calls take time (they are not instant). async/await tells JavaScript to wait for these operations to finish before moving to the next line — without freezing the server for other users. Every function that talks to a database must use await.

// Without async/await — runs instantly, gets nothing (empty promise)
const user = User.findById(id);     // ❌ Wrong!

// With async/await — waits for MongoDB to respond
const user = await User.findById(id); // ✅ Correct!
Middleware
Intermediate Request Processor
Express.js

A function that sits between the incoming request and the final controller. It has access to req, res, and next. It can modify the request, terminate it early (e.g., reject unauthorized users), or pass it on by calling next().

ES Modules
ECMAScript Module System
JavaScript

The modern JavaScript way to share code between files using import and export instead of the old require() and module.exports. We set "type": "module" in package.json to enable this for the entire project.

// Old way (CommonJS — DO NOT USE)
const express = require('express');

// New way (ES Modules — what we use)
import express from 'express';
ObjectId
MongoDB Unique Document Identifier
Database

Every document saved in MongoDB automatically gets a unique _id (e.g., 507f1f77bcf86cd799439011). This is a 24-character hex string called an ObjectId. When we store a reference to another document (like which user owns a transaction), we store their ObjectId. This is called a "relation" — linking two collections together.

populate()
Mongoose Join / Reference Resolver
Database

When a Transaction has category: ObjectId("abc123"), MongoDB only stores the ID, not the full category object. .populate("category") tells Mongoose to automatically fetch the full category document and replace the ID with the actual data — similar to a database JOIN in SQL.

Transaction.findById(id).populate("category", "name icon color");
// Instead of: { category: "abc123" }
// You get:    { category: { name: "Food", icon: "🍔", color: "#FF0" } }
fcmToken
Firebase Cloud Messaging Token
Push Notifications

A unique ID that Google Firebase assigns to each specific phone/device. Our backend stores this token for each user. When we need to send a push notification (like a budget alert), we send the message to Firebase along with the user's fcmToken, and Firebase delivers it to the right phone instantly.

mimeType
Multipurpose Internet Mail Extensions Type
File Upload

The standardized name for a file's format. When a receipt image is uploaded, the mimeType tells the backend exactly what kind of file it is. Examples: image/jpeg, image/png, application/pdf. We use this to reject non-image uploads (e.g., someone trying to upload a .exe file).

CORS
Cross-Origin Resource Sharing
Security

A browser security rule that blocks websites from making requests to a different domain. Since our mobile app (on localhost:3000) talks to our API (on localhost:5000), we must explicitly tell the backend to allow this using the cors middleware. Without it, browsers block every request.

Rate Limiter
Request Frequency Controller
Security

Limits how many requests one IP address can make in a time window. We allow 100 requests per 15 minutes for general routes, and only 10 per 15 minutes for login/register (to prevent brute-force password attacks). If exceeded, the API returns 429 Too Many Requests.

Authentication Flow

How users register, login, and access protected routes.

📝 Register Flow

1

Mobile App sends POST /api/v1/auth/register

Body: { name: "Mihir", email: "mihir@test.com", password: "pass123" }

↓
2

Joi validates the data

Checks: name has ≥2 chars? email is valid? password has ≥6 chars? If any fail → returns 400 error.

↓
3

Check if email already exists

User.findOne({ email }) — if found, returns 409 Conflict "Email already registered".

↓
4

bcryptjs hashes the password

The pre-save hook in User.js automatically converts "pass123" → "$2b$10$X7Vx8xK..." before saving.

↓
5

User saved to MongoDB + JWT generated

User.create({...}) saves the user. generateToken(user) creates a JWT containing the user's _id.

↓
6

Response: 201 Created

{ success: true, data: { user: {...}, token: "eyJhbG..." } } — The mobile app saves this token locally.

🔐 Protected Route Access Flow

After login, every request to a protected route follows this pattern:

// Mobile App sets this header on every request after login:
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJfaWQiOiI2NjcifQ.xZ8k...

// auth.middleware.js reads this and verifies it:
const decoded = jwt.verify(token, JWT_SECRET);  // cracks open the token
const user = await User.findById(decoded._id);   // gets user from DB
req.user = user;                                   // attaches user to request
next();                                            // lets controller run
⚠️ What about Logout?
Since JWTs are stateless (no session stored on server), logout is handled client-side — the mobile app simply deletes the saved token. Without the token, all future requests will fail the auth check automatically.
📌 Section 7

Database Concepts

Understanding Indexes, Virtuals, Hooks, and Enums in our Mongoose models.

⚡ Indexes — Speed Boosters

An index in MongoDB is like the index at the back of a textbook. Instead of reading every page to find a word, you jump directly to the right page. Without indexes, MongoDB reads every single record to find matches — which is extremely slow with large data.

// From Budget.js — these 3 indexes cover the 3 most common queries:
budgetSchema.index({ user: 1, category: 1, period: 1 }); // check duplicate budgets
budgetSchema.index({ user: 1, isActive: 1 });           // get active budgets
budgetSchema.index({ user: 1, startDate: -1 });         // sort newest first

// 1 = ascending sort, -1 = descending sort (newest first)

🔮 Virtuals — Calculated Fields

A Virtual is a property that looks like a database field but is never stored in MongoDB. It is calculated in real-time from other fields. This prevents data from going "out of sync."

// From SavingsGoal.js — calculated on-the-fly, not stored in DB
savingsGoalSchema.virtual("progressPercent").get(function() {
  return Math.min(100, (this.currentAmount / this.targetAmount) * 100);
});

savingsGoalSchema.virtual("remainingAmount").get(function() {
  return Math.max(0, this.targetAmount - this.currentAmount);
});

// If targetAmount = 10000 and currentAmount = 6000:
// progressPercent → 60 (auto-calculated, not stored)
// remainingAmount → 4000 (auto-calculated, not stored)

🪝 Pre-Save Hooks — Automatic Triggers

A Hook is a function that automatically runs at a specific moment. The pre("save") hook runs right before a document is saved to MongoDB — perfect for automatic transformations.

// From User.js — automatically hashes password before saving
userSchema.pre("save", async function(next) {
  if (!this.isModified("password")) return next(); // skip if not changed
  
  const salt = await bcrypt.genSalt(10);
  this.password = await bcrypt.hash(this.password, salt);
  next(); // continue saving
});

// From Category.js — auto-generate normalized name for duplicate checking
categorySchema.pre("validate", function(next) {
  this.normalizedName = this.name.trim().toLowerCase(); // "Coffee" → "coffee"
  next();
});

🔒 Enums — Strict Value Lists

An Enum restricts a field to only allow specific predefined values. This prevents invalid data from ever entering the database.

// Only these exact 3 strings are allowed for transaction type
const TRANSACTION_TYPES = ["income", "expense", "transfer"];

// Only these 4 are allowed for recurring intervals
const RECURRENCE_INTERVALS = ["daily", "weekly", "monthly", "yearly"];

// If app sends type: "random" → MongoDB rejects it automatically ❌
// If app sends type: "expense" → MongoDB accepts it ✅

AI Features Explained

How we integrated OpenAI into the backend — and how fallbacks protect the app when AI is unavailable.

🏷️
AI Categorization

When a transaction is created, we send the description and merchant name to OpenAI. It returns the most likely category (e.g., "Swiggy" → "Food & Dining"). Uses categorizer.service.js.

📊
Financial Insights

Analyzes the user's last 3 months of spending patterns and generates personalized advice like "You spend 40% more on weekends." Uses insights.service.js.

🔮
Budget Prediction

Looks at past 3 months of spending per category and suggests budget amounts for the next month. Uses predictor.service.js.

💬
Finance Chatbot

A conversational AI assistant that answers personal finance questions like "How much did I spend this month?" by reading the user's actual transaction data. Uses chatbot.service.js.

🛡️ Graceful Degradation (Fallback System)

✅ The App Never Crashes Without OpenAI
We built a two-tier system. If the OPENAI_API_KEY is missing or the API call fails, the system automatically falls back to keyword-based matching — not an error. 
// categorizer.service.js — simplified example
export const categorizeExpense = async (data) => {
  // Try AI first
  if (OPENAI_API_KEY) {
    const aiResult = await callOpenAI(data);
    return { category: aiResult, source: "ai" };
  }

  // Fallback to keyword matching if no API key
  const desc = data.description.toLowerCase();
  if (desc.includes("swiggy") || desc.includes("zomato"))
    return { category: "Food & Dining", source: "keyword" };
  if (desc.includes("uber") || desc.includes("ola"))
    return { category: "Transport", source: "keyword" };

  return { category: "Other", source: "default" }; // last resort
};
📌 Section 9

Background Cron Jobs

Automated tasks that run on a schedule — no user action required.

💡 What is a Cron Job?
A "cron job" is a task scheduled to run automatically at a specific time — like an alarm clock for your server. The word "cron" comes from the Greek word "chronos" (time). We use the node-cron library to define these schedules using a special 5-part cron expression.

⏰ Understanding Cron Expressions

// Format: MINUTE  HOUR  DAY-OF-MONTH  MONTH  DAY-OF-WEEK
"0 9 * * *"   // → Every day at 9:00 AM (9 o'clock, any day, any month)
"0 8 * * *"   // → Every day at 8:00 AM
"0 7 1 * *"   // → On the 1st day of every month at 7:00 AM
"0 * * * *"   // → Every hour (at minute 0)

📋 Our 4 Cron Jobs

FileScheduleWhat It Does
budgetAlert.job.js Daily 9 AM Checks all active budgets for all users. If spending has exceeded the alert threshold (e.g., 80%), it sends an in-app notification.
billReminder.job.js Daily 8 AM Looks for recurring transactions (Netflix, Rent, EMI) that are due within the next 2 days. Sends reminder notifications.
monthlyReport.job.js 1st of Month Generates a financial summary for every active user for the previous month and sends it as an email report.
currencySync.job.js Every Hour Fetches the latest exchange rates for INR, USD, EUR, GBP from an external API and caches them for fast access.

📦 How They Are Started

All cron jobs are started inside server.js after the server boots and MongoDB connects successfully:

// server.js — starts all jobs after MongoDB connects
const startCronJobs = async () => {
  const { initBudgetAlertJob }  = await import("./jobs/budgetAlert.job.js");
  const { initBillReminderJob } = await import("./jobs/billReminder.job.js");
  const { initMonthlyReportJob } = await import("./jobs/monthlyReport.job.js");

  initBudgetAlertJob();   // ⏰ starts the 9AM daily alarm
  initBillReminderJob();  // ⏰ starts the 8AM daily alarm
  initMonthlyReportJob(); // ⏰ starts the 1st-of-month alarm
};
📌 Section 10

Common Interview Questions

Ready-to-use answers you can give confidently in any interview.

Q1
What is the difference between SQL and MongoDB?

SQL (Relational): Stores data in fixed rows and columns like a spreadsheet. Strict schema — you cannot add a new field without altering the whole table. Examples: MySQL, PostgreSQL.

MongoDB (NoSQL): Stores data as flexible JSON-like documents. You can add new fields to individual documents without affecting others. This makes it great for apps that evolve quickly, like ours.

Q2
Why use JWT instead of sessions?

Sessions store user data on the server. For a mobile app with millions of users, you need to store millions of sessions — this doesn't scale well.

JWT is stateless. The user's info is encoded inside the token itself. The server doesn't store anything — it just verifies the token's signature. This makes it infinitely scalable and perfect for mobile apps and REST APIs.

Q3
What is Middleware and why do you use it?

Middleware is a function that processes a request before it reaches the controller. We use it to separate cross-cutting concerns — things that apply to many routes (like authentication, logging, and rate limiting) — from the actual business logic. This keeps controllers clean and avoids repeating code across every single route.

Q4
How do you store passwords securely?

We never store plain-text passwords. When a user registers, we use bcryptjs to generate a "salt" (random data) and hash the password. The hash is a one-way function — it cannot be reversed. When logging in, we hash the typed password again and compare it to the stored hash using bcrypt.compare().

Q5
What is the MVC pattern and do you follow it?

MVC stands for Model-View-Controller. We follow a similar pattern:
• Model = src/models/ — Data structure
• Controller = src/controllers/ — Business logic
• View = Not here (our "view" is the mobile app frontend)

We extend MVC with Routes, Middleware, and Services to handle the complexity of a production REST API.

Q6
What happens if OpenAI is down or the API key is missing?

The app uses Graceful Degradation. If the OpenAI API key is missing or the API call fails, the AI services automatically fall back to keyword-based matching. For example, if the description contains "Swiggy", we return "Food & Dining" without needing AI. The app never crashes — it just uses a less sophisticated method.

Q7
What is async/await and why is it important in Node.js?

Node.js runs JavaScript on a single thread. If a database query took 2 seconds and we waited (blocked), no other users could be served during those 2 seconds. async/await is JavaScript's way of saying "start this task, but don't wait here — serve other users, and come back when this task is done." This makes our backend handle thousands of simultaneous users efficiently.

Q8
Why did you use Joi instead of manual validation?

Manual validation of a single endpoint takes 40+ lines of if/else code and is easy to miss edge cases. Joi defines rules in 4 clean lines, automatically validates and transforms data (e.g., lowercase email, trim spaces), collects ALL errors at once (not just the first one), and makes rules instantly readable to any developer maintaining the code later.
